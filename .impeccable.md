# Design Context

## Design Context

### Users
Broad, mixed audience: indie developers running side projects, DevOps engineers monitoring production systems, and small business owners checking if their site is up. Experience levels vary widely. The common thread: **users open this app when something might be wrong** — the emotional context is mild anxiety, urgency, or routine vigilance. The interface must project calm and confidence immediately.

### Brand Personality
**Precise. Trustworthy. Frictionless.**

Clean & modern SaaS — polished and professional, sitting naturally alongside Linear, Vercel, or Resend. Not a developer toy and not enterprise bloat. Something a developer would be proud to show a non-technical client.

### Aesthetic Direction
- **Dark-mode-first**: Deep neutral backgrounds, sharp borders, precise typography. Light mode available as user preference.
- **Reference**: Linear / Vercel — minimal chrome, very high contrast text, tight spacing, sharp edges (avoid excessive rounding), restrained animation.
- **Anti-reference**: Avoid the Uptime Kuma original's "hacker energy" aesthetic. No neon accents, no glassmorphism, no bouncy animations.
- **Color**: Purple-primary (already established) used sparingly as a signal color, not decoration. Status colors (green/red/yellow) carry the most visual weight — they are the product.
- **Typography**: Clean, weight-varied hierarchy. Data density is a feature, not a bug.
- **Motion**: Purposeful only. State changes (status flips, data loading) deserve subtle animation. Background decoration does not.

### Design Principles
1. **Status is the hero** — Uptime, down/up indicators, and response times are the core value. They should always be the most visually prominent information on any screen.
2. **Clarity under pressure** — Users check this app when worried. Hierarchy must be instant and unambiguous. No hunting, no parsing.
3. **Progressive complexity** — Simple on first look. Power features available without clutter. Empty states teach; filled states stay out of the way.
4. **Earn every element** — No decorative cards, no filler copy, no redundant labels. Every visible element must serve a job.
5. **Dark-first, accessible always** — Dark mode is the primary experience. All semantic colors must maintain contrast in both modes. Color is never the only status signal.

## Design Tokens & Technical Reference

### Typography
- **Font**: Figtree (variable, sans-serif) with advanced feature settings: `cv02`, `cv03`, `cv04`, `cv11` for the sans stack; `ss02`, `zero` for mono.
- **Text rendering**: `optimizeLegibility` globally, with `opsz` variation set.
- **Scrollbar**: Thin, uses `--border` color on transparent track — unobtrusive.

### Color System (OKLCH)
All colors use OKLCH for perceptual uniformity.

**Primary (purple):**
- `--primary`: `oklch(0.546 0.245 262.881)` — used for CTAs, ring focus, charts
- `--primary-subtle`: `oklch(0.623 0.214 259.815 / 0.15)` — tinted backgrounds

**Status colors (carry the most visual weight):**
- `--success`: `oklch(0.596 0.145 163.225)` — green
- `--danger`: `oklch(0.577 0.245 27.325)` — red
- `--warning`: `oklch(0.828 0.189 84.429)` — amber

**Dark mode background scale:**
- `--bg`: `oklch(0.181 0.005 285.823)` — deepest base
- `--sidebar`: `oklch(0.21 0.006 285.885)` — slightly lighter
- `--navbar`: `oklch(0.190 0.006 285.885)` — nav surface
- `--overlay`: `oklch(0.210 0.006 285.885)` — modal/dialog backgrounds
- `--border`: `oklch(0.342 0.017 285.786)` — subtle but visible dividers

### Border Radius
- Base unit: `--radius-lg: 0.5rem` — all radii derive from this
- Sharp by default: `rounded-lg` on buttons (not pill/rounded-full unless `isCircle`)
- Consistent scale: `xs` → `4xl` all proportional to `--radius-lg`

### Component Patterns
- **Components**: React Aria Components (accessibility primitives) + `tailwind-variants` (variant system)
- **Button intents**: `primary`, `secondary`, `warning`, `danger`, `outline`, `plain`
- **Badge intents**: `primary`, `secondary`, `success`, `info`, `warning`, `danger`, `outline`
- **Touch targets**: `touch-hitbox` utility ensures 44×44px minimum tap area
- **Spacing**: Tight; prefers `sm:` breakpoint reductions (mobile-first with denser desktop)

### Interaction & Animation
- Library: `tw-animate-css` for utility-driven animations
- Motion: Only for state changes — no decorative or ambient animation
- Focus: `focus-visible` rings using `--ring` (purple), always offset from background
